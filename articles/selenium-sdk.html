Selenium-SDK: from JsonWire to W3C in D

May 2026

The problem: a dead dependency on a dead protocol

After two months building Insurgent around WebKit GTK, I learned the hard way that it does not run on Windows. The library is Linux-only, and the Windows port had been abandoned years ago. The project needed a cross-platform browser engine, and the obvious choice was Chrome via Selenium.

The obstacle was that D had no working Selenium bindings. The only existing option was selenium.d, last updated in 2018, built on the JsonWire protocol that Selenium had already deprecated. Modern ChromeDriver speaks W3C WebDriver. JsonWire requests were being ignored or rejected outright.

What JsonWire to W3C actually means

JsonWire and W3C WebDriver are not just different JSON wrappers around the same commands. The endpoint layout changed, the desired capabilities format changed, and the error response schema changed. In JsonWire, you POST to /session with a flat desiredCapabilities object. In W3C, the same endpoint expects a nested capabilities object with alwaysMatch and firstMatch arrays. A JsonWire session request against a modern ChromeDriver returns a 500 with no session ID, and every subsequent command fails.

The protocol gap was not a version mismatch. It was a different wire format, different capability negotiation, and different error handling. Every request path and every payload shape had to be revisited.

Session negotiation: the first W3C hurdle

The fix started at the entry point. Session creation now builds a W3C-compliant capabilities payload and parses the standardized response shape: 

Session startSession(string browser = "chrome")
{
    JSONValue capabilities;
    capabilities["alwaysMatch"] = [
        "browserName": browser,
        "acceptInsecureCerts": true,
    ];

    JSONValue payload;
    payload["capabilities"] = capabilities;

    auto response = request("/session", RequestMethod.POST, payload);
    string sessionId = response["value"]["sessionId"].str;
    return Session(sessionId, response["value"]["capabilities"]);
}

The response parser had to handle both the old JsonWire sessionId top-level field and the new W3C value.sessionId nested field. Auto-detection lets the library tolerate legacy drivers without a hard fork: 

string extractSessionId(JSONValue response)
{
    if (auto p = "sessionId" in response)
        return p.str;
    if (auto p = "value" in response)
        if (auto id = "sessionId" in *p)
            return id.str;
    throw new Exception("No session ID in response");
}

Modeling the API after Ruby

I had spent a lot of time with Ruby's Selenium bindings while building AutoHandshake, and the API felt natural. The D version follows the same conventions: a Driver owns the session, WebElement wraps node references, and locators are strongly typed instead of raw strings. 

struct By
{
    string strategy;
    string value;

    static By cssSelector(string value)
    {
        return By("css selector", value);
    }

    static By xpath(string value)
    {
        return By("xpath", value);
    }

    static By id(string value)
    {
        return By("id", value);
    }
}

Element lookup returns a WebElement that carries its own element ID, so the caller never manages opaque UUID strings directly: 

class WebElement
{
    private string _elementId;
    private Session _session;

    this(Session session, string elementId)
    {
        _session = session;
        _elementId = elementId;
    }

    void click()
    {
        _session.request(
            "/element/" ~ _elementId ~ "/click", RequestMethod.POST);
    }

    void sendKeys(string keys)
    {
        JSONValue payload;
        payload["text"] = keys;
        _session.request(
            "/element/" ~ _elementId ~ "/value", RequestMethod.POST, payload);
    }

    string text()
    {
        auto response = _session.request(
            "/element/" ~ _elementId ~ "/text", RequestMethod.GET);
        return response["value"].str;
    }
}

Driver auto-detection and lifecycle management

One quality-of-life feature from the Ruby ecosystem is that the bindings often find the driver executable for you. The D port replicates that with a path search across platform conventions: 

string findDriver(string name)
{
    version (Windows)
        name ~= ".exe";

    foreach (dir; environment.get("PATH").split(pathSeparator))
    {
        string candidate = buildPath(dir, name);
        if (exists(candidate))
            return candidate;
    }

    throw new Exception("Driver not found: " ~ name);
}

The Driver struct manages the child process and shuts it down cleanly on scope exit. A scope(exit) in the calling code is enough to guarantee cleanup, which fits D's RAII patterns: 

struct Driver
{
    Pid process;
    string executable;
    ushort port;

    static Driver start(string name = "chromedriver")
    {
        string exe = findDriver(name);
        ushort port = findFreePort();
        Pid pid = spawnProcess([exe, "--port=" ~ to!string(port)]);
        // Wait for the port to accept connections
        waitForPort(port);
        return Driver(pid, exe, port);
    }

    void stop()
    {
        if (process !is Pid.init)
        {
            kill(process, SIGTERM);
            wait(process);
        }
    }
}

Implicit waits and synchronization

A raw Selenium script without waits breaks immediately on slow-loading pages. The library exposes implicit-wait configuration on the session and wraps common patterns in retry loops with configurable timeouts: 

void setImplicitWait(Duration timeout)
{
    JSONValue payload;
    payload["ms"] = timeout.total!"msecs";
    request("/timeouts", RequestMethod.POST, payload);
}

WebElement findElement(By by, Duration timeout = 10.seconds)
{
    auto start = MonoTime.currTime;
    while (MonoTime.currTime - start < timeout)
    {
        try
            return findElementImpl(by);
        catch (Exception e)
            sleep(100.msecs);
    }
    throw new Exception("Element not found: " ~ by.value);
}

JavaScript execution and screenshot capture

Two features that Insurgent needed heavily were script injection and screenshot capture for debugging. Both map cleanly to W3C endpoints. executeScript takes a raw string and a variadic argument list, serializing each argument into the W3C element-reference format when needed: 

JSONValue executeScript(string script, JSONValue[] args ...)
{
    JSONValue payload;
    payload["script"] = script;
    payload["args"] = args;
    auto response = request(
        "/execute/sync", RequestMethod.POST, payload);
    return response["value"];
}

ubyte[] screenshot()
{
    auto response = request(
        "/screenshot", RequestMethod.GET);
    return Base64.decode(response["value"].str);
}

What is still missing

The library covers the core WebDriver surface that Insurgent needed: navigation, element interaction, form input, script execution, screenshots, and session management. It does not yet implement the Actions API for low-level mouse and keyboard input, nor does it support BiDi (WebDriver Bidi) for network interception or console log streaming. Remote Grid support and mobile WebDriverApp endpoints are also absent.

Those gaps are acceptable for now. The goal was not parity with the Java or Python bindings. The goal was a working W3C foundation in a language that had none.

What this project proved

  • A language without ecosystem coverage for "obvious" tools like browser automation becomes a hard sell for real-world projects the moment you need cross-platform UI interaction.
  • Protocol migration is tedious but mechanical. The hard part is deciding which legacy behavior to preserve and which to drop.
  • API ergonomics matter more than endpoint coverage. A clean findElement(By.css("...")) surface is worth more than every obscure WebDriver endpoint exposed raw.
  • D's compile-time metaprogramming and RAII patterns make for a very pleasant binding: zero-cost abstractions, deterministic cleanup, and no garbage-collection surprises during long-lived browser sessions.
  • One working library can unblock an entire category of projects. Insurgent could not have shipped on Windows without this.
